git# Plan: Rootless Podman + systemd (Quadlet) deployment, no Podman Compose

Replaces `podman compose` orchestration with rootless Podman managed by systemd
(via Quadlet), for both local libvirt dev VMs and production/cloud VMs, using
the same `provision-vm.yml` playbook. Includes a TLS-terminating reverse proxy
in front of a production static frontend build, with a self-signed certificate
(no Let's Encrypt/ACME).

## Decisions made

- Orchestration: Podman Quadlet units (`.container` / `.network` / `.volume`),
  generated by the systemd generator into rootless `--user` services. No
  `podman generate systemd`, no Podman Compose.
- Image delivery to the VM: build on the control host → `podman save` → copy
  tarball to the VM → `podman load` into `zedr`'s rootless store. No registry.
- Quadlet unit definitions: Jinja2 templates checked into the playbook
  directory, rendered by `ansible.builtin.template` so secrets/paths can be
  injected.
- Frontend in production: real production build. `npm run build` produces
  static assets; no Vite dev server at runtime.
- Reverse proxy: Caddy container terminates TLS on 80/443; backend and
  frontend no longer publish host ports.
- TLS certificate: self-signed, generated once by the playbook. No ACME/Let's
  Encrypt.
- Static asset delivery: multi-stage Containerfile — stage 1 builds the
  frontend with `npm run build`, stage 2 is a Caddy image with `dist/`, the
  cert, and the Caddyfile baked in. There is no separate frontend container;
  `mojesaldoo-frontend.container` is replaced entirely by
  `mojesaldoo-proxy.container`.

## Target topology

| Unit | Image | Published ports | Notes |
|---|---|---|---|
| `mojesaldoo.network` | — | — | shared network, name-based DNS between containers |
| `mojesaldoo-pgdata.volume` | — | — | Postgres data, replaces compose's `pgdata` volume |
| `mojesaldoo-db.container` | `postgres:16` | none | `Notify=healthy` + `HealthCmd` (pg_isready) |
| `mojesaldoo-backend.container` | `mojesaldoo-backend:latest` | none | gunicorn; reachable only on the internal network; `After=`/`Requires=mojesaldoo-db.service` |
| `mojesaldoo-proxy.container` | `mojesaldoo-proxy:latest` (was `frontend`) | `80:80`, `443:443` | Caddy + baked-in static `dist/`; TLS via self-signed cert; proxies `/api/*` to `backend:8000`; `After=mojesaldoo-backend.service` |

`postgres:16` is pulled from Docker Hub at first start (public base image, not
a private registry) unless the VM must be fully offline, in which case it can
be saved/loaded the same way as the app images.

## Work items

### 1. Containerfile / build changes
- [ ] `automation/containers/frontend/Containerfile`: rewrite as multi-stage.
      Stage 1 (`node`): `npm ci` + `npm run build`. Stage 2 (`caddy`): copy in
      `dist/`, the self-signed cert/key, and a `Caddyfile`. Update the
      healthcheck to hit `https://127.0.0.1/` (or an unauthenticated path)
      instead of the old Vite dev-server check.
- [ ] New `automation/containers/frontend/Caddyfile`: reverse-proxy `/api/*`
      to `backend:8000`; serve everything else as static files from the
      baked-in `dist/` directory; `tls /certs/cert.pem /certs/key.pem`
      (explicit cert paths so Caddy never attempts ACME).
- [ ] `automation/containers/backend/Containerfile`: no functional change
      required — it already doesn't declare `EXPOSE` as a host binding
      concern; just confirm nothing assumes a host-published port.

### 2. Quadlet unit templates
New directory `automation/vms/playbooks/templates/`:
- [ ] `mojesaldoo.network.j2`
- [ ] `mojesaldoo-pgdata.volume.j2`
- [ ] `mojesaldoo-db.container.j2` — image, network, volume mount, env file,
      health check, `Notify=healthy`
- [ ] `mojesaldoo-backend.container.j2` — image, network, env file, bind
      mounts for `data/media` and `data/logs` (`:Z,U`), `After=`/`Requires=`
      on the db unit
- [ ] `mojesaldoo-proxy.container.j2` — image, network, published ports
      80/443, cert dir bind mount (read-only), `After=` on the backend unit

Rendered to `~zedr/.config/containers/systemd/` via `ansible.builtin.template`
with `become_user: zedr`.

### 3. Secrets and certificate generation (playbook tasks)
- [ ] Generate a Postgres password and `DJANGO_SECRET_KEY`, following the
      existing idempotent pattern already used for the `zedr` login password

- [ ] Write these to `0600` env files under something like
      `/home/zedr/.config/mojesaldoo/*.env`, owned by `zedr`, referenced via

- [ ] Generate a self-signed TLS cert/key using the `community.crypto`
      collection (`openssl_privatekey`, `openssl_csr`, `openssl_certificate`

- [ ] Add `community.crypto` to `automation/vms/requirements.yml`.

### 4. Image build + transfer

- [ ] Copy the tarball to the VM (`ansible.builtin.copy` or `synchronize`).
